荷塘悦色经营智能系统架构

这是一张系统架构总览图，用来表达荷塘悦色经营智能系统如何围绕本地经营事实组织语义理解、能力路由、执行链路和反馈回路。重点是主链清晰、边界明确，而不是交付责任或部署说明。

确定性经营真相层
语义操作系统
Capability Graph
Safe Execution
Analysis Worker
Semantic Quality Loop

系统主链

Text → Conversation Control → Semantic Intent → Early Stop Gate → Lane Select → Capability Graph → Serving Semantic Layer → Safe Execution → Reply Guard → Answer / Action

查询链：快、稳、可审计
分析链：深度复盘、建议
反馈回路：失败样本、评审、升级
1

上游业务系统层

原始业务事实来源，不做 AI 判断。

Source

POS / 收银系统

订单、结算、支付、反结等交易事实。

Source

会员系统

会员档案、余额、标签、券包快照、沉默状态。

Source

技师系统

当前在岗、上钟明细、项目承接、提成结构。

Source

充值 / 流水接口

充值本金、赠送、账户变动、耗卡与异常交易。

Source

营销 / 团购渠道

来源、转化、活动触达、会员渠道归因。

Source

外部情报 / 弱信号

平台规则、环境、天气、周边变化线索。

2

确定性经营真相层

所有重要经营判断都最终落回这里，AI 不改写事实。

这里是本地经营真相层：原始业务事实先进入本地库，再形成派生指标、日报、顾客画像和 serving 读面。所有高价值经营判断都必须能回落到这个层。
结构化经营数据不进入知识库作为主真相源；它们继续走 PostgreSQL、指标语义层与 Safe Execution。
Kernel

数据接入 client

对接上游 API，按门店拉取原始事实。

Kernel

数据标准化 normalize

统一字段口径，保留 rawJson 证据。

Kernel

数据同步 sync

营业日切分、ready 状态、回补控制。

Truth

PostgreSQL

事实层、派生层、控制面与审计面统一落库。

Truth

原始事实表

消费、充值、流水、技师快照、上钟和营销明细。

Truth

派生指标 / serving views

日报、7日、30日、画像、风险信号和读面视图。

3

语义操作系统层

把“人话问题”变成“可识别、可路由、可执行的经营意图”。

总入口主链
Input

用户提问 / 文本输入

企微、Bridge、Query API 等统一入口。

Control

会话控制

补槽、上下文继承、对象切换、topic switch。

Intent

语义意图识别

meta / query / analysis 三大入口。

Gate

早停判断

闲聊、澄清、不支持问题、边界说明。

State

会话语义状态

lastEffectiveText、补槽继承、显式重置。

查询链路
Lane

查询 / 分析分流

query lane 进入快链路。

Graph

能力图谱

能力选择、fallback、owner path。

Plan

查询计划

实体、时间、指标、维度、回答形态收口。

Serve

语义服务层

query plan 映射到 serving / runtime render。

Audit

语义执行审计

失败样本、执行轨迹、质量闭环回灌。

异步分析链路
Lane

异步分析链路

analysis lane，适合深度复盘、诊断、建议。

Worker

Analysis Worker

较重异步分析，不阻塞主查询链路。

Fact

事实注入

日报、7日、30日、画像、环境和弱信号。

Output

诊断 / 建议输出

重点问题、原因、优先动作和复盘结论。

4

执行与调度层

查询执行、异步分析、权限边界与调度编排。

Execution

语义服务层

承接 query plan，组织受控读面与 runtime render。

Execution

SQL 编译器

受控生成查询，不让模型自由拼底层 SQL。

Trust

安全执行

权限、字段边界、门店范围、审计与口径控制。

Runtime

Query Engine

负责 serving / runtime render / analysis 结果总装。

Worker

Scheduled / Analysis Worker

一个负责同步报表投递，一个负责异步诊断任务。

5

输出与协同层

结果展示与协同交付，面向 HQ、店长、运营等业务角色。

Delivery

Bridge / 外部消息入口

企业微信、网关、消息回调的统一入口适配。

Delivery

Query API / 只读接口

对外提供受控读面，不暴露底表与自由查询。

Delivery

报表 / HQ 总览 / 顾客经营

日报、周报、月报、5店总览、顾客经营输出。

Delivery

结果回复 / 动作建议

直接答复、诊断建议、执行提示和后续协同动作。

6

反馈闭环层

把失败样本、复盘发现、路由观测重新回灌到语义主链。

语义失败样本

未识别问法、边界误判、入口掉线、clarify 失配。

复盘发现

nightly review、人工评审、执行后观察、doctor 暴露。

路由观测

shadow compare、lane 漂移、quality backlog 聚类。

升级积压项

把问题收敛成 capability / prompt / route 的明确改造项。

回灌升级

回写到语义意图识别、会话语义状态和能力图谱。

AI 边界

AI 只增强理解、归因、解释、复盘与质量回路，不直接充当经营真相，不绕过安全执行主链。

数据库 / 指标语义层 / 知识检索层

结构化经营数据走 PostgreSQL 与指标语义层；知识检索层只承载 SOP、规则、口径、培训资料。

当前仍是事实缺口

等位 / 候钟实时状态、待结账 / 待结算实时单据、优惠券完整发券-核销-回店闭环。
