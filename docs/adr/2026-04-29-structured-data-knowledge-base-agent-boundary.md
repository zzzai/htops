# ADR: Structured Data, Knowledge Base, And Agent Boundaries

日期：2026-04-29  
状态：accepted

---

## 背景

`htops` 已经走到一个容易架构漂移的阶段。

一方面，项目正在强化 AI-native 方向：

- `Semantic Intent`
- `Capability Graph`
- `AI lanes`
- `analysis worker`
- `semantic quality loop`

另一方面，仓库里沉淀的门店经营数据、顾客经营数据、技师履约数据、充值和消费事实，天然会让人产生一个错误冲动：

> “既然都要接大模型，那干脆把所有数据做成知识库，让模型去搜。”

对连锁足疗/门店经营这种场景，这条路是错的。

核心问题不在“是否使用知识库”，而在于：

**必须先把结构化经营数据、规则知识、以及 Agent 编排边界分清楚。**

---

## 决策

### 1. 经营流水数据不做知识库真相源

以下数据默认不进入知识库作为主真相层：

- 营业额、实收、订单数、客单价、充值、退款、优惠
- 顾客消费明细、充值明细、储值余额、生命周期状态
- 技师上钟记录、技师业绩、点钟率、加钟率、提成
- 门店排行、技师排行、复购率、沉默会员、团购转化

这些数据必须继续走：

`业务库 / 分析库 / PostgreSQL truth layer / 指标语义层 / safe execution`

原因：

1. 这些数据是结构化事实，不适合靠向量检索做主查询。
2. 这些数据需要统一口径、权限控制和可审计执行。
3. 这些数据的价值在于聚合、比较、排序、归因，不在于文本相似度检索。

### 2. 知识库只承载非流水类业务知识

知识库适合承载：

- 门店运营 SOP
- 会员制度、充值规则、退款规则
- 技师提成规则、培训手册
- 项目介绍、禁忌说明、服务流程
- 投诉处理流程
- 营销活动方案
- 指标口径解释
- 数据字段含义说明

换句话说：

**知识库回答“规则是什么、流程是什么、口径是什么、应该怎么做”；**
**数据库/语义层回答“昨天发生了什么、谁排第一、哪家下滑、哪些客户在流失”。**

### 3. Agent 负责理解、编排、分析，不直接碰底层事实

Agent 的职责定义为：

1. 理解自然语言问题
2. 识别用户角色与权限
3. 决定调用哪类工具：
   - 指标查询
   - 顾客查询
   - 技师查询
   - 知识库检索
   - 报表/图表生成
4. 汇总结构化结果与知识检索结果
5. 生成经营解释、动作建议和输出文案

Agent 不直接承担：

- 自由拼 SQL
- 越权查底表
- 直接把所有事实塞进向量库再召回
- 绕过 `Capability Graph` 和 safe execution 主链

### 4. 项目采用“数据库 / 指标语义层 / 知识库 / Agent”的组合架构

项目目标架构明确为：

```text
用户提问
   ↓
Semantic Intent / 权限 / 上下文
   ↓
Capability Graph / Agent 调度
   ↓
┌───────────────────────┬───────────────────────┐
│ 结构化指标 / 查询工具 │ 知识库检索工具        │
│ PostgreSQL truth layer│ SOP / 规则 / 口径 / FAQ │
└───────────────────────┴───────────────────────┘
   ↓
Safe Execution / Answer / Action
```

其中：

- 结构化经营数据走 PostgreSQL + 指标语义层
- 规则知识走知识库
- 大模型 / Agent 负责理解问题、编排工具、生成分析结论

### 5. 指标语义层优先级高于“直接让模型写 SQL”

当用户问：

> 昨天各门店营业额排名  
> 最近 30 天复购率下降的门店有哪些  
> 哪些会员快流失了  

系统优先走：

`自然语言 -> Query DSL / Capability -> 指标服务/查询服务 -> 安全 SQL -> 结果 -> 大模型解释`

而不是：

`自然语言 -> 大模型直接拼 SQL -> 直接打数据库`

### 6. 所有“可以问业务数据”的 AI 能力都必须走权限与审计

无论是老板、店长、运营、技师，任何数据问答都必须具备：

- 角色识别
- 门店范围限制
- 字段级边界
- 查询与结果审计

这条边界同样适用于知识库工具和数据库工具。

## 结果

### 正面影响

1. 项目架构更清晰，不会把“知识库”误当成经营数据真相层。
2. 后续 AI 增强方向更明确：重点做 `Semantic Intent + Capability Graph + Agent 编排 + 分析解释`，而不是把所有明细都向量化。
3. 指标口径、权限审计、safe execution 继续保持在主链中心。
4. 知识库的价值也被明确下来：承载规则、流程、口径、培训，而不是订单明细。

### 负面影响

1. 不能用“一个向量库解决所有问题”的偷懒方案。
2. 需要持续建设指标语义层和结构化查询工具。
3. 数据仓库/分析库、知识库、Agent 三条线要分别治理，工程面更严谨。

---

## 对当前 htops 的直接约束

### 应继续强化

- `src/client.ts`
- `src/normalize.ts`
- `src/store.ts`
- `src/query-plan.ts`
- `src/capability-graph.ts`
- `src/query-engine.ts`
- `src/app/analysis-service.ts`

### 不应做的事

- 不把 `fact_*` / `mart_*` 数据全文塞进知识库做主查询
- 不让模型直接拼接任意 SQL
- 不把“制度 / 规则 / FAQ 检索”与“经营事实查询”混成一个工具
- 不把“问答方便”凌驾于权限审计之上

---

## 一句话收束

`htops` 的正确方向不是“纯知识库问答”，而是：

**`数据库 / 指标语义层 / 知识库 / Bounded Agent / 权限审计` 的组合架构。**
