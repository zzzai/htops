# Hetang Ops 四阶段推进路线图

## 1. 目标

当前项目不追求一步到位做“全能经营分析机器人”，而是按业务价值和系统稳定性分 4 个阶段推进：

1. 先稳定出数
2. 再安全可查
3. 再单店深度分析
4. 最后总部横向与深度智能化

这样做的原因很直接：

- 门店每天先要有稳定日报
- 权限和审计必须先收紧
- 深度分析必须建立在真实字段和稳定 mart 之上

---

## 2. 当前总体状态

截至 `2026-03-31`：

- 5 店 `2026-03` 的 mart 日指标和日报已基本覆盖
- 其余 4 店不是“没拉到”，而是和义乌店一样，已大体落到 PostgreSQL
- 仍需继续盯的重点不是“从 0 到 1”，而是“补齐缺口、稳定发送、把分析真正接进企微主链路”

当前最需要被盯住的一项异常：

- `锦苑店` 最近一次回补存在 `1.4 fetch failed`

---

## 3. 阶段一：稳定出数与按店归属

### 3.1 目标

让 5 家店每天都能稳定同步、稳定归属、稳定生成日报。

### 3.2 当前已完成

- PostgreSQL 已替代 SQLite
- `org_id -> canonical store_name` 归属已落地
- 营业日 `05:00` 截止已落地
- 5 店 `2026-03` mart 日指标 / 日报已大体落库
- 定时调度已落地：同步、build、send 三段任务

### 3.3 当前未收口

- 某些事实表仍有个别门店缺 `2026-03-31` 或少量边界天
- `锦苑店` 存在一次 `1.4` 局部失败
- 日报发送状态落库还未稳定形成闭环

### 3.4 阶段一的完成标准

- 5 店过去 7 天 `mart_daily_store_metrics` 连续完整
- 5 店过去 7 天 `mart_daily_store_reports.complete=true`
- 5 店过去 7 天 `send_status` 正常回写
- 无持续性 `partial` 门店

### 3.5 下一步动作

1. 按周切片把 `2026-03` 回补再跑稳一轮
2. 单独复核 `锦苑店` 的 `1.4`
3. 补齐日报发送状态回写观察

当前阶段状态：`进行中，接近收口`

---

## 4. 阶段二：企微安全查询与问答治理

### 4.1 目标

让员工在企微里“能查、只能查该查的、查得有审计、查得有限额”。

### 4.2 当前已完成

- `whoami / report / sync / status` 权限体系已落地
- 单店 / 多店 / 总部角色已支持
- 限额和审计已落地
- 自然语言日报类查询已支持
- `whoami` 已按门店名展示，不再只显示 OrgId

### 4.3 当前未收口

- 需要把真实企微人员批量导入成正式权限清单
- 需要继续扩充自然语言查询覆盖面
- 企微“偶发不回复”仍需继续压到最低

### 4.4 阶段二的完成标准

- 真实员工权限清单全部导入
- 店长只能看到自己门店或授权门店
- 总部人员可看全部门店
- `/hetang whoami` 对所有已授权员工可稳定返回
- 日报类自然语言查询在企微可稳定命中和回复

### 4.5 下一步动作

1. 完成正式权限名单导入
2. 把企微不回复问题继续收敛到“有日志、可定位、可恢复”
3. 把自然语言入口从“日报类”扩到“日报 + 若干核心指标问答”

当前阶段状态：`已基本可用，仍需稳定化`

---

## 5. 阶段三：单店店长深度复盘

### 5.1 目标

把 CrewAI sidecar 做成“店长真正能用”的单店复盘助手，先服务单店经营决策。

### 5.2 当前已完成

- CrewAI sidecar 已独立部署
- sidecar 当前读取 PostgreSQL，不读实时 API
- 提示词已融合真实字段能力
- 已接入团购支付、会员支付、钟型、加钟、项目类型等原始字段摘要

### 5.3 当前未收口

- 模型调用偶发超时
- 企微内还不能直接调用 CrewAI 复盘
- 还未形成“查询日报 -> 一键触发单店复盘”的主链路

### 5.4 阶段三的完成标准

- 义乌店近 7 天复盘可稳定生成
- 至少 1 家店长能实际拿到可执行建议
- 企微中可触发“单店近 7 天复盘”
- 输出严格遵守当前字段边界，不乱下结论

### 5.5 下一步动作

1. 给 CrewAI sidecar 接稳定模型
2. 固化“义乌店近 7 天店长复盘”
3. 再把这个能力挂到 OpenClaw 工具调用链

当前阶段状态：`进行中`

---

## 6. 阶段四：总部多店横向与深度智能化

### 6.1 目标

在单店稳定后，再做总部视角的横向对比、异常归因和深度分析。

### 6.2 当前尚未交付

- 总部多店横向对比版 CrewAI 工作流
- 企微内任意自然语言深度经营分析
- 更完整的办卡 / 充值归因闭环
- 财务级副项 / 类目映射

### 6.3 阶段四的完成标准

- 总部可在企微查询多店对比结论
- 至少具备营收、钟效、会员、技师 4 大横向对标维度
- 能解释“为什么差”，而不是只罗列数字

### 6.4 下一步动作

1. 在单店提示词稳定后，扩出总部版提示词
2. 补齐多店对比所需汇总和排名 mart
3. 再考虑把更开放的自然语言深度分析接进企微

当前阶段状态：`未开始，排在单店稳定之后`

## 8. 当前结论

如果按业务价值排序，最值得继续推进的不是“再做更多概念功能”，而是：

1. 先把阶段一彻底收口
2. 再把阶段二稳定到可放心给真实员工使用
3. 然后用阶段三证明“这个 AI 助手真的能帮店长做决策”

只有前三阶段跑稳了，第四阶段的总部级深度分析才有意义。
