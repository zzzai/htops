# Store Manager WeCom Bot Quick Guide Design

日期：2026-04-20
状态：approved
范围：店长在企微中使用 Hermes-htops 查询门店数据的速查教程

## 背景

当前已经有一份偏“日报/周报怎么读”的店长教程，但还缺一份更前置、更实用的文档：

- 店长在企业微信里到底该怎么问
- 用什么问法最稳
- 哪些问题用 `/hetang report`
- 哪些问题用 `/hetang query`
- 哪些需要进入 `reactivation` 命令面

这类文档不应该写成系统说明书，而应该写成“拿起企微就能照着发”的速查手册。

## 设计目标

新文档必须满足：

1. 店长第一次看也能马上照着问
2. 只写真实可用的当前命令面，不写理想状态
3. 明确区分“查数字”“问原因”“看名单”“回写结果”
4. 让店长知道查到以后该怎么动作，而不只是会发命令
5. 手机端阅读友好，篇幅短，结构直接

## 设计方向

采用 `速查手册`，不是 `完整产品手册`。

文档聚焦“日常高频使用”，不讲：

- 架构
- 模型
- 服务部署
- 技术排障

文档只讲：

- 在企微里怎么发
- 发什么最有效
- 哪种场景用哪条命令

## 内容结构

### 1. 什么时候用

按店长日常节奏写：

- 早上：看昨日结果
- 中午：查今天需要推进的会员跟进
- 晚上：复盘异常

### 2. 最稳的问法

明确推荐顺序：

1. `最稳`：`/hetang report`
2. `问原因/建议`：`/hetang query`
3. `查召回执行`：`/hetang reactivation summary|tasks|update`
4. `自然语言`：只作为补充，不作为第一推荐

### 3. 店长最常问的 6 类问题

固定成 6 组：

- 日报
- 指标快查
- 风险和建议
- 周报/周图
- 高价值待唤回名单
- 跟进结果回写

每组都给：

- 推荐发法
- 不推荐发法
- 适用场景

### 4. 问完以后怎么动作

把结果翻译成动作模板：

- 客流掉了
- 加钟率掉了
- 点钟率掉了
- 新增会员弱
- 储值弱

### 5. 常见错误

集中写 4 类高频误区：

- 没带门店
- 没带日期
- 一次问太多件事
- 想用模糊自然语言直接问复杂诊断

## 准确性边界

文档必须显式遵守当前实际能力边界：

- 单店权限店长可以直接用 `/hetang report`，默认查绑定门店
- 多店权限人员必须显式带门店
- 群聊自然语言需要 `@机器人`
- 自然语言命中更依赖“门店 + 日期 + 指标/日报”组合
- 复杂原因分析优先使用 `/hetang query`

## 验收标准

完成后，至少满足：

1. 店长看完前 1 分钟就知道第一句该怎么问
2. 文档中所有命令都能映射到当前实际命令面
3. 文档同时覆盖“查数”“问原因”“查名单”“回写”
4. 文案完全站在店长视角，没有技术术语负担
