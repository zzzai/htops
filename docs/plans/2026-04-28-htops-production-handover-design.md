# htops 正式生产交付文档设计

## 目标

产出一份可直接交给外部接手方的正式生产交付文档，使其在自带上游 API 与企微通道的前提下，能够理解 `htops` 的定位、部署边界、运行方式、改造点、验收方式与运维要求，并在自己的服务器上独立完成生产部署。

## 读者

- 技术负责人 / 架构负责人
- 后端开发 / 实施工程师
- 运维 / DevOps

## 文档策略

采用单一主文档，不拆散成多份附件。原因：

1. 交付场景以混合读者为主，需要一份材料同时讲清业务定位、系统边界、部署步骤与运维方式。
2. 对外移交时，多附件更容易丢失上下文，导致负责人、开发、运维各自只看到局部。
3. `htops` 当前已经不是单点脚本，而是一个由 PostgreSQL、Node worker、Python 查询 API、可选 bridge 组成的模块化单体系统，交付说明需要统一叙述。

## 文档结构

主文档将采用以下结构：

1. 项目介绍
2. 交付范围与责任边界
3. 生产部署形态
4. 环境与依赖要求
5. 目录结构与关键文件
6. 上游 API 对接要求
7. 企微通道对接要求
8. 部署步骤
9. 配置说明
10. 服务启动与 systemd
11. 验收清单
12. 日常运维
13. 常见故障与排查
14. 升级、回滚、备份恢复
15. 交接清单

## 抽象边界

对外交付文档不保留当前现网接口编号、现网历史异常样本、现网敏感目标地址等痕迹。文档只保留：

- 通用上游 API 数据域要求
- 通用同步与报表产物要求
- 可替换的消息发送契约
- 可执行的正式生产部署步骤

## 交付原则

1. 让负责人先看懂“项目是什么、接手后要负责什么”。
2. 让开发知道“如果不复用当前 API 协议，具体改哪些模块”。
3. 让运维能直接照着完成安装、systemd、健康检查、上线与回滚。
4. 明确哪些部分是必须组件，哪些是可选组件。
5. 不夸大成熟度，明确当前系统的生产前提与数据质量责任边界。
