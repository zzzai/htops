# Hetang Night Window Optimization Design

## Context

`03:00-04:00` 是当前唯一稳定可用的 API 访问窗口。现状里夜间任务存在两个核心问题：

1. `syncStores()` 仍按“核心接口 / 技师接口 / 会员流水”粗阶段执行，不能保证所有门店先尽快补齐 `1.1/1.2/1.3/1.5/1.6/1.7/1.8`。
2. `1.4` 仍按 `Type 1..11 x 全量会员卡` 全扫描，核心实验门店会在窗口内被拖死。

## Goal

把夜间窗口改成“先保证高价值原始事实，再做会员流水补强”的执行策略：

1. 固定门店顺序：`迎宾店 -> 义乌店 -> 华美店 -> 锦苑店 -> 园中园店`
2. 固定接口波次：`1.1 -> 1.2 -> 1.3 -> 1.5 -> 1.6 -> 1.7 -> 1.8 -> 1.4`
3. `1.4` 改为候选会员卡驱动，只拉近 30 天活跃或变动过的卡
4. 其余门店历史回补优先收敛最近 30 天缺口；迎宾店保留后续向更长历史扩展的路径

## Chosen Design

### 1. Runtime orchestration

`HetangOpsRuntime.syncStores()` 不再按大阶段跑整店，而是按单接口波次跨店推进：

1. 全部门店跑 `1.1`
2. 全部门店跑 `1.2`
3. 全部门店跑 `1.3`
4. 全部门店跑 `1.5`
5. 全部门店跑 `1.6`
6. 全部门店跑 `1.7`
7. 全部门店跑 `1.8`
8. 最后再按固定门店顺序尝试 `1.4`

这样可以把“原始主事实闭环率”放到第一优先级，避免某一家门店的 `1.4` 拖慢其它门店的主链路。

### 2. Candidate-card driven user-trade sync

`1.4` 不再默认遍历全部会员卡，而是支持通过 `syncPlan.selectedCardIds` 精确指定候选卡集合。

候选卡来源：

1. 近 30 天 `fact_recharge_bills` 中出现过的 `CardId` / `CardNo`
2. 近 30 天 `fact_consume_bills` 中出现过的 `HandCardCode` / `HandCardCodes`
3. `fact_member_cards_current.raw_json` 里的近期 `LastUseTime`

候选卡按“最近活动时间倒序”排序。若候选集为空，则 `1.4` 本次直接跳过，不再退化成空 `Id` 的全店查询。

### 3. Access-window budget control

`1.4` 启动前按候选卡数量估算耗时；若剩余可用窗口不足，则标记该店为 `user trades deferred`，不再硬闯窗口。

### 4. Backfill priority

覆盖率驱动的夜间补数保留现有框架，但优先级调整为：

1. 其它 4 店优先最近 30 天缺口
2. 迎宾店先补最近 30 天，再为更长历史保留继续回补的路径
3. 大店 `1.4` 仍然不参与全量卡历史扫描

## Error Handling

1. 单店异常仍只影响本店，不阻断其它门店
2. 单接口同步依旧复用 `syncHetangStore()` 的错误记录逻辑
3. 当只执行单个技师接口时，内部不应再触发无意义的 15 秒技师冷却

## Testing

1. `runtime.test.ts`
   - 锁定新的波次顺序与固定门店顺序
   - 验证 `1.4` 最后执行，且仅向同步器传入候选卡
2. `sync.test.ts`
   - 验证 `selectedCardIds` 只会触发指定会员卡请求
   - 验证空候选集不会退化成 11 次空 `Id` 查询
   - 验证单技师接口执行时不会额外触发技师冷却
