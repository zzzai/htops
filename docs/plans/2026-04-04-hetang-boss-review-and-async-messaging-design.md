# Hetang Boss Review And Async Messaging Design

## Goal

让 `hetang-ops` 更稳地理解老板口吻问题，并把异步复盘的等待体验做得更像真实助手。

## Context

当前链路已经有两套能力：

- 单店、长时间窗、宽口径问题会进入异步深度复盘
- 跨店总部问题会进入 `hq_portfolio` 的同步汇总分析

但两处还有明显缺口：

- “近30天盘子稳不稳” 这类口语化问法，识别覆盖还不够稳
- 异步复盘虽然已有“阶段进度 / 完成摘要”，但文案更像系统提示，不够自然

## Options

### Option A: 所有老板口吻问题统一进异步分析

优点：

- 规则最简单
- 输出深度最统一

缺点：

- 会把本来可以秒回的总部跨店风险类问题全部拖进长流程
- 企微体感会更慢

### Option B: 保持双路架构，只补触发词和文案层

优点：

- 改动小，风险低
- 单店深度复盘继续异步，跨店总部总览继续同步
- 不影响现在已经稳定的主路径

缺点：

- 需要更仔细地维护“单店异步 / 跨店同步”的边界

### Option C: 再加一层新的老板问题编排器

优点：

- 逻辑看起来更“统一”

缺点：

- 架构增重
- 与现有 `query-intent`、`analysis-router`、`inbound` 重叠
- 这阶段属于很典型的“为了优雅先把系统搞复杂”

## Recommendation

选择 Option B。

原因很简单：这条路最稳，也最符合当前项目节奏。老板问单店长期走势时，异步深度复盘有价值；老板问“五店谁最危险”时，同步回答反而更符合现场使用习惯。系统要像一个懂业务的助手，而不是一个逢问题必排队的流程爱好者。

## Design

### 1. 老板口吻识别

- 扩展 `query-intent.ts` 的 HQ 口语触发词，覆盖：
  - `近30天盘子稳不稳`
  - `最近盘子稳不稳`
  - `本月经营怎么样`
  - `哪家店最近最危险`
  - `最近哪家店掉得厉害`
- 保持现有边界：
  - 单店、范围型、宽口径问题继续走异步分析
  - 跨店、总部视角问题继续走 `hq_portfolio`

### 2. 异步阶段提示

- 保持现有状态不变：
  - `created`
  - `reused-pending`
  - `reused-running`
  - `reused-completed`
- 升级用户文案：
  - 明确当前阶段
  - 明确正在做什么
  - 明确完成后会先给摘要，再补完整内容

### 3. 完成回推摘要

- 保持 `runtime.ts` 当前“完成摘要 + 正文”结构
- 在头部增加更自然的完成提示：
  - 先说已经看完
  - 再给一句完成摘要
  - 最后补正式回复
- 继续兼容 `notify.ts` 的分段逻辑，不破坏企微拆段

## Testing

- `query-intent.test.ts`
  - 增加新的 HQ 口语问题覆盖
- `analysis-router` 相关测试
  - 验证“近30天盘子稳不稳”会进异步分析
- `inbound.test.ts`
  - 验证新文案会被第一时间回给企微
- 新增 `analysis-queue-message.test.ts`
  - 验证排队中 / 分析中 / 已完成复用三种文案
