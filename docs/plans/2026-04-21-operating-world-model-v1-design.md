# Operating World Model v1 Design

日期：2026-04-21
状态：approved
范围：把 `htops` 的经营智能系统进一步提升为“可推演的经营世界模型”初版

## 一、定义

这里的 `Operating World Model v1`，不是视频生成、3D 空间生成意义上的世界模型。

在 `htops` 里，它的定义是：

**一个以真实经营事实为底座，能够持续构造“门店经营世界状态”，并对动作后果做有边界推演、解释与学习的经营认知模型。**

它要回答的不是：

- “能不能像人一样聊天”

而是：

- 这家店现在处在什么经营状态
- 为什么会这样
- 如果采取某个动作，最可能发生什么
- 哪些变量最值得优先盯
- 这次动作到底有没有效果

---

## 二、为什么现在要做这层抽象

当前 `htops` 已经具备：

- 经营事实层
- 门店环境层
- 画像 / 分层 / 召回
- 日报 / 周报 / 企微查询
- 夜间 review / semantic quality / doctor 学习闭环

这说明系统已经不缺“零散能力”，缺的是：

**一个能把多层状态、机制、动作、结果统一起来的中枢认知框架。**

如果没有这层抽象，后续会出现 4 个问题：

1. 数据越来越多，但系统只是“知道很多”，不是真懂
2. 外部知识、竞对、行业、宏观信号无法和门店内部经营放在同一个推演面
3. 画像/分层/召回会停留在增长局部，不会上升到全局经营决策
4. AI 很会解释，但不会形成稳定、可复验、可学习的世界状态模型

所以，`Operating World Model v1` 的目标，是把当前系统从：

- `数据系统 + 问答系统 + 召回系统`

升级为：

- `状态系统 + 推演系统 + 决策系统 + 学习系统`

---

## 三、备选方案

### 方案 A：继续把能力分散在日报、周报、画像、外部知识里

优点：

- 改动最小
- 短期交付最快

缺点：

- 无统一状态建模
- 无统一动作推演
- 无法把总部诊断、店长动作、召回策略收成同一套认知体系

### 方案 B：在现有 Operating Intelligence 基础上，增加 World Model 层

核心思路：

- 保留现有 owner path 和事实层
- 新增“世界状态快照 + 机制库 + 场景推演 + 决策出口 + 学习回灌”
- 不引入第二套 runtime，不把 LLM 变成真相源

优点：

- 与现有 `htops` 架构最兼容
- 风险最低
- 可分阶段落地
- 能逐步变聪明，而不是推倒重来

缺点：

- 初期仍然是 bounded world model，不是完全自演化系统

### 方案 C：直接做 AI-native 自治经营模拟器

优点：

- 理论上长期上限高

缺点：

- 风险太大
- 难以保证事实边界
- 与当前仓库的 Ontos-lite 边界不兼容

## 四、推荐方案

选择 **方案 B**。

原因很明确：

- 当前项目已经有足够多的真实经营 owner modules
- 当前最缺的是统一世界状态，而不是更多零散功能
- 需要的是“在现有真相层上增加推演层”，不是新起一套 AI 平台

---

## 五、核心架构

`Operating World Model v1` 由 5 层组成。

### 1. 世界状态层

这一层负责回答：

- 系统当前认为“世界是什么样”

它由 4 类状态拼起来：

#### A. customer_state

- 客户价值
- 消费习惯
- 到店时段偏好
- 技师偏好
- 沉默风险
- 召回可行性

#### B. store_state

- 门店店型
- 房间和承接能力
- 供给结构
- 项目结构
- 价格带
- 晚场能力
- 当前经营健康度

#### C. market_state

- 商圈结构
- 竞对压力
- 停车便利
- 周边餐饮/酒店/办公/住宅
- 天气/节气/节假日
- 门店事件

#### D. industry_state

- 行业景气
- 平台流量变化
- 城市消费变化
- 用工与技师供给
- 宏观消费情绪
- 资本与赛道热度

这 4 类状态必须以不同 truth level 构成，不能混为一谈。

### 2. 经营机制层

这一层负责回答：

- 哪些状态之间存在经营机制关系

它不是自由发挥，而是受控机制库，例如：

- 晚饭后 + 夜生活商圈 + 停车便利 -> 晚场承接机会升高
- 客流没掉但储值掉了 -> 转化或信任链条有问题
- 高价值沉默会员 + 熟悉技师仍在岗 -> 召回成功概率上升
- 竞对加大团购强度 -> 价格敏感客更容易流失
- 深夜需求上升但供给不足 -> 承接缺口导致营收损失

机制层是 `world model` 的关键，不是只存数据，而是存“世界如何运转”。

### 3. 场景推演层

这一层负责回答：

- 如果做某个动作，最可能发生什么

初版只做 3 类 bounded simulation：

#### A. explanation simulation

解释当前结果可能由哪些因素共同造成。

例如：

- 最近晚场走弱，不只是客流下降，也可能是供给不足 + 节气转冷 + 周边竞对分流

#### B. counterfactual simulation

做轻量“如果...会怎样”的推演。

例如：

- 如果把晚场容量提高 15%，当前高价值沉默客的承接会不会改善
- 如果继续主推储值，是否会牺牲客流恢复

#### C. action simulation

针对候选动作，给出收益、风险、依赖条件。

例如：

- 动作：晚饭后重点召回熟悉技师老客
- 可能收益：预约率提升
- 风险：当日供给不足会造成承接失败
- 前提：技师班次和房态需要先确认

初版不追求高精度预测器，而追求：

- 有边界
- 可解释
- 能辅助经营动作排序

### 4. 决策出口层

这一层负责把世界模型结果转成业务可用输出：

- 店长日报提示
- 总部周报判断
- 门店经营诊断
- 召回动作建议
- 风险预警
- 资源配置建议

也就是说，`world model` 不是独立报告系统，而是：

- 服务现有日报
- 服务现有周报
- 服务现有 query / analysis
- 服务现有 customer growth

### 5. 学习回灌层

这一层负责让世界模型越来越准。

它的输入包括：

- 查询误判
- 分析解释偏差
- 召回反馈结果
- 策略是否有效
- 哪些假设被证明无效

它的输出包括：

- 机制修正候选
- 样本集候选
- backlog 候选
- 部署后复验项

## 六、状态、机制、动作、结果的关系

这是 `Operating World Model v1` 的核心闭环：

```text
世界状态 -> 机制判断 -> 候选动作 -> 场景推演 -> 动作执行 -> 结果反馈 -> 世界状态更新
```

对应到当前项目：

- 世界状态：store/customer/market/industry snapshots
- 机制判断：deterministic rules + bounded AI synthesis
- 候选动作：日报动作、周报动作、召回动作、总部指导动作
- 场景推演：HQ diagnosis / store advice / reactivation prioritization
- 动作执行：企微、门店跟进、召回回写、经营动作落地
- 结果反馈：feedback snapshots / nightly review / semantic quality
- 世界状态更新：next-day state rebuild

## 七、数据边界

### 1. 硬事实

可直接进入状态核心与严谨算分：

- 内部交易
- 内部会员
- 技师
- 排钟
- 房态
- 到店
- 反馈事实

### 2. 软事实

可进入世界状态，但对动作影响必须 bounded：

- 商圈结构
- 竞对强弱
- 天气
- 节气
- 停车便利
- 门店事件

### 3. 弱信号

可进入 industry_state，但默认不进入单客主评分：

- 宏观消费环境
- 平台规则变化
- 行业热度
- 资本热度
- 赛道趋势

这是世界模型可信的前提。

---

## 八、AI 在 World Model 里的位置

AI 不是世界模型本身，而是世界模型的增强器。

### 1. AI 负责

- 上下文组织
- 机制候选解释
- 动作建议候选
- 反事实叙事
- 多因素诊断摘要
- 学习回灌结构化

### 2. AI 不负责

- 伪造事实
- 直接替代硬事实算分
- 绕过 bounded rule shell
- 无审计修改世界状态

### 3. 初版原则

- deterministic state shell
- bounded AI hypothesis layer
- auditable decision outputs

---

## 九、初版要服务的高价值场景

### 1. 门店经营诊断

例如：

- 为什么这家店最近客流掉了
- 是商圈、竞对、供给还是转化问题

### 2. 召回与营销动作优先级

例如：

- 当前该优先召回哪类人
- 当前更适合打预约还是打即时到店

### 3. 总部周报与资源配置

例如：

- 哪些门店是经营问题
- 哪些门店是位置/竞对问题
- 哪些门店该追加资源

### 4. 风险预警

例如：

- 客流未掉但储值掉
- 高价值客户回流弱
- 夜场承接不足
- 竞对压力上升

---

## 十、和现有架构的关系

`Operating World Model v1` 不是一个新系统，而是现有架构上的一层升级。

它应建立在：

- `src/store.ts`
- `src/store-query.ts`
- `src/customer-growth/*`
- `src/store-external-context.ts`
- `src/customer-growth/environment-context.ts`
- `src/app/reporting-service.ts`
- `src/query-engine.ts`
- `src/app/semantic-quality-service.ts`
- `src/app/conversation-review-service.ts`
- `src/ops/doctor.ts`

之上。

明确约束：

1. 不引入第二套 ontology runtime
2. 不把 `src/runtime.ts` 变成世界模型总控
3. 不让 world model 绕开 capability graph 和 safe execution
4. 不让 AI 直接替代事实层

---

## 十一、v1 的边界

`v1` 不做这些：

- 高自由度自治 agent 经营系统
- 完整强化学习经营模拟器
- 自动改策略并自动上线
- 无边界世界自演化

`v1` 要做的是：

- 先把世界状态收敛清楚
- 先把机制库和 bounded simulation 做起来
- 先让诊断、决策、指导、学习说同一种语言

---

## 十二、结论

`Operating World Model v1` 的本质，不是让系统更像“聊天 AI”，而是让系统更像：

**一个能够理解门店经营世界、能做全局思考、能做有边界推演、并能从结果中持续学习的经营认知模型。**

这会成为 `htops` 从“经营工具集合”升级为“经营智能系统”的关键中层。
