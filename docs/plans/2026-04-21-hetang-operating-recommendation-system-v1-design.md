# Hetang Operating Recommendation System v1 Design

日期：2026-04-21
状态：approved
范围：把推荐系统与 LLM 的通用思想翻译成荷塘悦色可落地的“经营动作推荐系统”

## 一、定义

这里不做互联网内容平台意义上的推荐系统。

在 `htops` 里，`Hetang Operating Recommendation System v1` 的定义是：

**一个以门店经营事实、需求时钟、供给约束和外部环境为底座，为客户运营、门店经营和总部管理持续推荐“下一步最优动作”的经营推荐系统。**

它要推荐的不是：

- 你下一条该看什么视频
- 你下一件该买什么商品

它要推荐的是：

- 哪个客户今天该不该联系
- 该在什么时段联系
- 该约哪个时段到店
- 该推什么项目 / 技师 / 权益
- 哪家店今天该抓哪个时段
- 哪家店该补供给、控让利、补会员沉淀
- 总部下周该把资源投到哪家店、哪条链路

所以，这套系统本质上是：

**Next Best Action Recommendation**

而不是内容推荐。

---

## 二、为什么现在要做这层抽象

当前项目已经有：

- 顾客画像 / 分层 / 召回
- 到店时段分布
- 时段偏好与触达窗口
- 门店日报 / 周报
- 世界模型
- 外部观察与门店主数据设计
- Hermes / 企微问答与经营诊断

这些能力说明系统已经具备很多“推荐前置条件”，但还没有形成统一的推荐框架。

当前缺口主要有 5 个：

1. 候选动作还比较分散，没有统一 candidate 抽象
2. 排序仍然偏局部，更多是召回排序，不是全经营动作排序
3. LLM 现在更多做解释，没有被严格放进推荐流水线的合适位置
4. 总部、门店、客户三个决策面还没有统一到一套 action recommendation 上
5. 反馈学习存在，但还没有正式成为推荐系统的训练闭环

如果不把这层抽象补上，后面会继续出现：

1. 数据越来越多，但动作仍然靠经验切换
2. 外部数据、评论、竞对、商圈、环境数据难以真正进入经营动作排序
3. LLM 容易变成“会说”，而不是“会推荐对的动作”
4. 世界模型会知道很多状态，但不会稳定产出最优动作

因此，这一轮不是去做“更大的模型”，而是要把系统从：

- `状态分析系统`

推进到：

- `经营动作推荐系统`

---

## 三、推荐系统里的 `item` 在荷塘到底是什么

这一步是整个设计最关键的地方。

在荷塘里，推荐系统的 `item` 不是内容，而是：

### 1. 面向客户运营的动作候选

例如：

- 今天联系某高价值沉默客
- 约周五晚饭后回店
- 推荐熟悉技师
- 推荐某项目升级
- 推荐轻储值 / 大储值
- 不联系，先观察

### 2. 面向门店经营的动作候选

例如：

- 主抓晚饭后承接
- 夜场放大高价值老客预约
- 午后控低效让利
- 调整某班次技师排布
- 优先补点钟承接
- 优先补新增会员后的 48 小时回访

### 3. 面向总部管理的动作候选

例如：

- 哪家店优先补资源
- 哪家店优先做夜场打法
- 哪家店优先修会员沉淀
- 哪家店优先修供需错配
- 哪条策略值得扩到 5 店复制

所以，荷塘推荐系统的核心不是：

- `给用户推荐内容`

而是：

- `给经营系统推荐动作`

## 四、备选方案

### 方案 A：继续用规则系统 + 人工判断

优点：

- 简单
- 上线快
- 可解释

缺点：

- 无法处理越来越复杂的状态组合
- 规则数量会膨胀
- 很难形成持续学习
- 无法让 LLM 和小模型各司其职

### 方案 B：混合式经营推荐系统

结构：

- 规则和 SQL 负责候选生成
- 小模型和统计模型负责主排序
- LLM 负责特征增强、候选重排、流程控制和解释
- 反馈数据负责学习闭环

优点：

- 与当前项目最兼容
- 风险最低
- 最符合你当前数据量和业务复杂度
- 可逐步变强，不需要一步到位重构

缺点：

- 初期不是端到端大模型
- 需要较严格的数据分层和流程边界

### 方案 C：直接做端到端生成式推荐系统

优点：

- 理论上上限高

缺点：

- 当前数据和反馈密度不够
- 很难控制事实边界、延迟和稳定性
- 容易把“经营动作排序”误做成“LLM 聊天式建议”

## 五、推荐方案

选择 **方案 B**。

一句话总结：

**荷塘现在最优解是“世界模型驱动的混合式经营推荐系统”：规则产候选，小模型做主排序，LLM 做增强重排与解释，反馈闭环做学习。**

## 六、总体架构

`Hetang Operating Recommendation System v1` 建议分成 6 层。

### 1. 候选生成层

职责：

- 先生成一批值得考虑的动作候选

候选来源：

- 高价值沉默客
- 需求时钟中的强需求时段
- 供需错配预警
- 门店异常指标
- 总部资源配置规则
- 外部环境变化

这个阶段必须 deterministic。

不能让 LLM 从空气里“生动作”。

### 2. 主排序层

职责：

- 对候选动作做第一轮硬排序

输入：

- 客户价值
- 到店概率
- 时段需求强度
- 承接可行性
- 毛利优先级
- 风险约束

这一层应由小模型、统计模型和规则约束共同完成。

### 3. LLM 增强层

职责：

- 从评论、门店资料、外部环境、竞对、事件中补充侧信息
- 生成结构化补充特征

这一层本质是：

- `LLM for feature engineering`

而不是直接决定最终推荐。

### 4. LLM 重排层

职责：

- 对 top-K 候选动作进行复杂语义判断和重排

适用问题：

- 多个动作分数接近时，谁更合适
- 外部环境有没有隐藏风险
- 某店某时段是否需要更谨慎

这一层对应：

- `PALR`
- `Chat-Rec`
- `GPT4Rec`

但只适合在 top-K 小范围做，不适合全量跑。

### 5. 解释生成层

职责：

- 把推荐翻译成总部、店长、客服可执行的话

例如：

- 为什么今天先抓晚场
- 为什么这个客人今天适合约周五夜场
- 为什么某家店今天不该扩大触达

### 6. 反馈学习层

职责：

- 根据动作执行结果修正下一轮推荐

核心反馈：

- 联系
- 回复
- 预约
- 到店
- 加钟
- 储值
- 复购
- 毛利质量

## 七、系统核心公式

内容推荐常写成：

`F(User, Item, Context) -> Preference`

荷塘经营推荐系统更准确的表达是：

```text
F(Actor, ActionCandidate, WorldState, Constraints) -> ActionPriority
```

其中：

- `Actor`
  - 客户运营
  - 门店店长
  - 总部管理者

- `ActionCandidate`
  - 某个具体动作

- `WorldState`
  - 客户状态
  - 门店状态
  - 需求时钟
  - 承接时钟
  - 外部环境

- `Constraints`
  - 营业时间
  - 房态
  - 技师供给
  - 风险边界
  - 利润边界

`ActionPriority` 则决定：

- 该不该做
- 先做哪个
- 应该以什么方式做

---

## 八、主排序应该由什么完成

推荐系统最容易犯的错误，是把 LLM 当主排序器。

对荷塘当前阶段，主排序应该坚持：

### 1. 规则负责硬约束

例如：

- 门店已满房，不允许放大该时段触达
- 顾客短周期刚来，不应该高频打扰
- 深夜非营业窗口，不产生该类动作

### 2. 小模型负责概率判断

建议优先使用：

- Logistic Regression
- GBDT / XGBoost 风格树模型
- 分层贝叶斯 / 分层统计模型

预测目标：

- 到店概率
- 预约概率
- 加钟概率
- 储值概率
- 供需错配风险

### 3. 需求时钟负责时间上下文

提供：

- 哪个时段需求偏强
- 哪个时段更适合动作
- 哪个时段存在供需错配

### 4. 世界模型负责状态拼装

把：

- 客户
- 门店
- 外部环境
- 反馈学习

统一成可读的世界状态。

## 九、LLM 在这套推荐系统里的正确位置

这里必须非常明确。

### 应该做的 4 件事

#### 1. 特征工程

把原始文本、评论、外部资料、门店信息转成结构化 side information。

例如：

- 用户评论主题
- 竞对活动标签
- 商圈夜生活强度描述
- 小红书种草 / 吐槽点

#### 2. 流程控制

让 LLM 充当 controller，负责：

- 调工具
- 调 SQL
- 调 world model
- 拼接证据
- 输出推荐说明

#### 3. 候选重排

只在 top-K 候选动作之间做复杂判断。

#### 4. 推荐解释

把复杂模型输出翻译成经营语言。

### 不应该做的 3 件事

#### 1. 不直接定义真相

LLM 不能覆盖 confirmed facts。

#### 2. 不直接替代主排序

LLM 不应全量给所有动作打分后直接输出 top1。

#### 3. 不直接越过供给约束

即使 LLM 觉得某动作“语义上合理”，也不能忽略营业时间、房态、技师和风险约束。

## 十、如何映射你给的这些论文

### 1. `Recformer`

可借鉴点：

- 把序列表示做强
- 用文本/属性理解 item

在荷塘的翻译：

- 未来可把客户行为、项目、技师、门店事件、时段、外部环境序列化做统一表征

当前阶段不推荐直接上。

### 2. `PALR`

可借鉴点：

- 候选集合 + 用户画像 + LLM prompt -> rerank

在荷塘的翻译：

- 用 LLM 对 top-K 经营动作做重排

这是当前可以落地的。

### 3. `Chat-Rec`

可借鉴点：

- LLM 可交互、可解释、可根据上下文做推荐

在荷塘的翻译：

- Hermes / 企微 / 总部问答场景里的经营动作解释器与建议器

### 4. `RecMind`

可借鉴点：

- Planning
- Memory
- Tools

在荷塘的翻译：

- LLM controller + personalized memory + world knowledge + tool use

这非常适合当前 Hermes-agent 演化方向。

### 5. `Meta Generative Recommender`

可借鉴点：

- 长期架构方向

当前不适合直接落地，原因：

- 数据规模不匹配
- 成本不匹配
- 反馈速度不匹配

## 十一、推荐系统的输出面

这套系统至少服务 3 个出口。

### 1. 客服 / 召回执行面

输出：

- 今天先打谁
- 用什么方式联系
- 推荐约哪个时段
- 哪个技师优先匹配
- 哪个客户先别打扰

### 2. 门店店长面

输出：

- 今天先抓哪个时段
- 哪个时段需求没接住
- 哪个动作最该做
- 哪个动作现在别做

### 3. 总部管理面

输出：

- 哪家店是资源优先级最高
- 哪家店的推荐打法值得复制
- 哪家店需要先控风险再求增长

---

## 十二、推荐系统的评估指标

不能照搬互联网内容推荐的指标。

内容平台通常看：

- CTR
- 完播率
- 时长
- 留存

荷塘经营推荐系统应重点看：

### 1. 动作命中

- 联系率
- 回复率
- 预约率
- 到店率

### 2. 结果质量

- 加钟率
- 储值率
- 复购率
- 客单提升

### 3. 经营结果

- 营收提升
- 毛利质量
- 高价值客户回流
- 供需错配改善

### 4. 推荐质量

- 推荐可执行率
- 推荐误打扰率
- 推荐解释一致性
- 推荐稳定性

### 5. 约束安全

- 是否打爆供给
- 是否伤害高价值客户体验
- 是否出现明显错误推荐

---

## 十三、与现有系统的映射

### 1. 与需求时钟的关系

需求时钟负责回答：

- 哪个时段需求更强
- 哪个时段更该抓

它是推荐系统的时间上下文层。

### 2. 与世界模型的关系

世界模型负责提供：

- 完整的经营状态
- 机制关系
- bounded simulation

它是推荐系统的状态底座。

### 3. 与门店主数据 / 外部观察层的关系

门店主数据和外部观察层负责提供：

- 门店事实
- 商圈
- 竞对
- 评论
- 环境

它们是推荐系统的 side information 和约束来源。

### 4. 与现有召回链路的关系

当前已有：

- `reactivationPriorityScore`
- `strategyPriorityScore`
- `executionPriorityScore`

未来推荐系统不应推翻这条链，而是：

- 先把它升级成客户运营动作推荐子系统
- 再逐步扩展到门店经营与总部管理动作推荐

---

## 十四、当前阶段最优落地顺序

推荐按 4 步走。

### 第一步：先做客户运营动作推荐

范围最小、反馈最清晰、最容易看到效果。

### 第二步：扩到门店经营动作推荐

例如：

- 哪个时段补供给
- 哪个时段控让利
- 哪个时段主抓会员沉淀

### 第三步：扩到总部资源推荐

例如：

- 哪家店先补资源
- 哪条打法先复制

### 第四步：引入 LLM top-K 重排和 controller

只在基础主排序稳定后再做。

---

## 十五、结论

荷塘真正应该做的，不是互联网意义上的“推荐系统照搬”，而是：

**世界模型驱动的经营动作推荐系统。**

它的正确架构是：

1. `规则 / SQL` 负责候选生成
2. `小模型 + 统计模型` 负责主排序
3. `需求时钟 + 供给约束 + 世界模型` 负责上下文与限制
4. `LLM` 负责特征增强、重排、流程控制、解释
5. `反馈学习` 负责持续校正

这条路线的好处是：

- 贴合当前 5 店数据规模
- 贴合线下门店物理约束
- 不会把 LLM 用错位
- 能一步步往更强的经营智能演进
